Fundamentos de Git

¿Qué es Git?

Git es un sistema de control de versiones distribuido que permite a los desarrolladores:

  • Rastrear cambios en el código
  • Colaborar en proyectos
  • Mantener un historial completo de versiones
  • Revertir cambios cuando sea necesario

Conceptos Clave

Repositorio Carpeta que contiene todo el historial del proyecto
Commit Snapshot de cambios con mensaje descriptivo
Rama Línea de desarrollo independiente
Merge Fusión de cambios de una rama a otra

Ciclo de Trabajo Básico de Git

GitHub: Colaboración en la Nube

¿Qué es GitHub?

GitHub es una plataforma web que aloja repositorios Git en la nube y añade características colaborativas:

  • Hosting de repositorios Git
  • Control de acceso y permisos
  • Pull Requests para revisión de código
  • Issues para seguimiento de problemas
  • Integración continua (CI/CD)
  • Colaboración en equipo

Flujo de Colaboración

  1. Fork: Copia un repositorio a tu cuenta
  2. Clone: Descargas el código a tu máquina
  3. Branch: Creas una rama para tu trabajo
  4. Commit: Haces cambios y commiteás
  5. Push: Envías cambios a tu fork
  6. Pull Request: Solicitas que revisen e integren tus cambios

Flujo Git + GitHub

Git Flow: Estrategia Profesional de Branching

Git Flow es una estrategia de branching introducida por Vincent Driessen que facilita la organización del ciclo de vida del desarrollo de software. Es ideal para proyectos que necesitan mantener versiones estables en producción mientras se desarrollan nuevas funcionalidades simultáneamente.

✅ Beneficios de Git Flow

  • Organización Clara: Diferentes ramas para diferentes propósitos
  • Control de Versiones: Manejo de múltiples versiones en paralelo
  • Flujo Controlado: Integración y pruebas antes de producción
  • Reducción de Riesgos: Código estable en main
  • Trabajo En Equipo: Menos conflictos, mejor coordinación

🎯 Casos de Uso Ideales

  • Proyectos con versiones públicas
  • Equipos de desarrollo grandes
  • Software que requiere estabilidad en producción
  • Aplicaciones con ciclos de release planeados
  • Necesidad de hotfixes rápidos en producción

El Flujo Visual de Git Flow

Tipos de Ramas en Git Flow

🔴 main (Producción)

Rama Principal

Propósito: Rama principal siempre lista para ser desplegada en producción.

Características:

  • Solo contiene versiones estables del software
  • Cada commit debe representar una versión liberada
  • Los commits están etiquetados con números de versión
  • Nunca se trabaja directamente en esta rama
Comandos típicos: git checkout main
git merge release/1.0

🟢 develop (Integración)

Rama de Integración

Propósito: Rama de integración donde convergen todas las características.

Características:

  • Punto de integración para todas las features
  • Código en constante evolución pero estable
  • Base para crear ramas de release
  • Debe estar siempre compilable y funcional
Comandos típicos: git checkout develop
git merge --no-ff feature/nueva-funcionalidad

🔵 feature/* (Características)

Rama Temporal

Propósito: Desarrollar nuevas características o mejoras.

Características:

  • Se crea desde develop
  • Una rama por característica
  • Se elimina después de fusionarla
  • Convención: feature/nombre-descriptivo
Flujo típico: git checkout -b feature/login develop
# ... hacer cambios ...
git checkout develop
git merge --no-ff feature/login
git branch -d feature/login

🟠 release/* (Versión)

Rama de Lanzamiento

Propósito: Preparar una nueva versión para producción.

Características:

  • Se crea desde develop cuando está lista una versión
  • Aquí se corrigen bugs menores y se actualizan versiones
  • Se fusiona en main y develop
  • Convención: release/v1.2.0
Flujo típico: git checkout -b release/v1.2.0 develop
# ... pruebas y arreglos ...
git checkout main
git merge --no-ff release/v1.2.0
git tag v1.2.0

🔴 hotfix/* (Corrección)

Rama de Emergencia

Propósito: Arreglar bugs críticos en producción rápidamente.

Características:

  • Se crea desde main (última versión en producción)
  • Se usa solo para arreglos críticos
  • Se fusiona en main y develop
  • Convención: hotfix/nombre-del-bug
Flujo típico: git checkout -b hotfix/crash-payment main
# ... arreglar bug ...
git checkout main
git merge --no-ff hotfix/crash-payment
git tag v1.2.1

Comparativa de Ramas

Rama Surge de Se fusiona en Duración Nombre Propósito
main - - Permanente main Producción estable
develop - - Permanente develop Integración de features
feature develop develop Temporal (días/semanas) feature/* Nueva funcionalidad
release develop main + develop Temporal (días) release/* Preparar versión
hotfix main main + develop Temporal (horas) hotfix/* Arreglo urgente
